# Phase 9 APEX Plan

## Analysis

### Project Structure

- `backend/` is a Laravel API with controllers, form requests, resources, observers, queued jobs, and PHPUnit feature/unit tests.
- `frontend/` is a Vue 3 + Vitest SPA using typed API modules, composables, page/view components, and Tailwind-based layout primitives.
- Phase tracking already exists in [`docs/phases/phase9.md`](/home/gerald/zeronoshow/docs/phases/phase9.md) and [`docs/dev/phase9.md`](/home/gerald/zeronoshow/docs/dev/phase9.md).

### Existing Patterns

- Backend behavior changes around reservations are centralized in [`backend/app/Observers/ReservationObserver.php`](/home/gerald/zeronoshow/backend/app/Observers/ReservationObserver.php).
- Business-scoped auth APIs live in [`backend/routes/api.php`](/home/gerald/zeronoshow/backend/routes/api.php) with simple controller authorization checks.
- SMS sending is modeled through `SmsLog` plus queued jobs like `SendVerificationSms` and `SendReminderSms`.
- Frontend API access uses `apiClient` wrappers and composables such as `useReservations`, `useWaitlist`, `useVoiceSettings`.
- Dashboard and reservation detail are the current user-facing anchors for customer context.

### Related Files

- Backend core: `Business`, `Customer`, `Reservation`, `ReservationObserver`, `ReservationController`, `CustomerController`, `routes/api.php`, `routes/web.php`, `routes/console.php`.
- Frontend core: `Dashboard.vue`, `ReservationDetailPage.vue`, `ReservationForm.vue`, `NavBar.vue`, `router/index.ts`, `types/reservations.ts`.
- Tests to mirror: reservation store/update tests, waitlist observer tests, voice settings tests, Leo/voice frontend component tests.

### Dependencies

- Laravel queues and scheduler are already in use.
- Sanctum auth and feature tests are already wired.
- Frontend uses Vitest + Vue Test Utils.
- `gh` CLI is available for PR/CI workflow.

### Current Architecture

- Customer data is currently minimal and phone-centric.
- Reservation creation returns customer reliability data but not CRM tags.
- Reservation status transitions trigger side-effects through the observer.
- No customer listing API exists yet beyond lookup.
- No reputation/review infrastructure exists yet.

### Key Constraints

- Keep changes aligned with existing Laravel/Vue conventions; avoid unrelated refactors.
- Phase 9 requires both backend and frontend coverage plus CI/PR completion.
- The current frontend structure differs slightly from the phase doc, so integration must target actual pages/components in the repo.
- Reservation blacklist behavior in the detailed spec is warning-based, not a hard block.

### Open Questions

- None blocking. Implementation will follow the detailed phase spec where it conflicts with the lighter tracking doc.

### Implementation Order

1. Add failing backend tests for CRM update, blacklist warning, review service/job, review observer, review settings, filters, and expiration command.
2. Implement migrations, enums, models, resources, requests, services, jobs, policies, routes, and controller changes until backend tests pass.
3. Add failing frontend tests for CRM panel/banner and reputation UI.
4. Implement frontend API/types/composables/views/components and integrate with existing pages/navigation until frontend tests pass.
5. Update phase tracking docs, run validation suites, perform adversarial review, fix issues, then commit/push/PR/CI-watch.

### Acceptance Criteria

- [ ] Customer CRM fields are persisted, editable, and returned through API resources.
- [ ] Reservation flows expose blacklist warning metadata without blocking creation.
- [ ] Review requests can be configured, scheduled after `show`, sent via SMS, clicked through tracked redirects, listed, and expired.
- [ ] Customer listing supports VIP, blacklist, and birthday-month filters.
- [ ] Frontend exposes CRM controls, blacklist warnings, customers filtering, and reputation dashboard pages.
- [ ] Backend and frontend automated tests pass locally.
- [ ] Phase tracking docs are updated.
- [ ] Branch is pushed, PR is created, and CI ends green.

### Risks

- HIGH: Phase 9 spans a large backend/frontend surface; type/resource drift can create avoidable test churn.
- MEDIUM: The phase doc references some non-existent frontend screens, so adaptation to the current app layout must stay consistent.
- MEDIUM: Scheduler/job semantics around delayed review sending need tests that assert intent without becoming flaky.
- LOW: Public redirect and review-link formatting can be implemented cleanly with existing patterns.
