# Phase 5 APEX Plan

## Analysis

### Project Structure
- `backend/` is a Laravel 12 API with Sanctum auth, versioned routes, Stripe/Twilio webhooks, queue jobs, observers, policies, form requests, JSON resources, Pest tests, and PostgreSQL-style migrations.
- `frontend/` is a Vue 3 + TypeScript SPA with Pinia, Vue Router, Axios API client, composition API composables, Tailwind-based design system, and Vitest component/page tests.
- `docs/phases/phase5.md` is the full specification. `docs/dev/phase5.md` is a task tracker and audit log; both show Phase 5 as not started.

### Existing Patterns
- Backend uses `HasUuids`, form requests with French messages, JSON resources, controller-level orchestration, dedicated services for provider APIs, and observer-driven async jobs.
- Stripe integration already exists via `StripeService` and `StripeWebhookController`; Phase 5 should extend those flows instead of replacing them.
- Frontend uses typed API helpers in `src/api`, stateful composables in `src/composables`, page-level data loading inside Vue SFCs, and Vitest with stubbed child components.

### Related Files
- Backend hotspots: `backend/routes/api.php`, `backend/app/Models/Business.php`, `backend/app/Observers/ReservationObserver.php`, `backend/app/Http/Controllers/Webhook/StripeWebhookController.php`, `backend/app/Services/StripeService.php`.
- Frontend hotspots: `frontend/src/router/index.ts`, `frontend/src/pages/Dashboard.vue`, `frontend/src/components/NavBar.vue`, `frontend/src/stores/auth.ts`, `docs/graphics/templates/template_site_vitrine.html`.

### Dependencies
- Existing: Stripe PHP SDK, Twilio SDK, Sanctum, Redis queues/caching, Vue Router, Pinia, Axios, Vitest.
- Required by Phase 5 but not yet present in code: Gemini function-calling integration, Telegram webhook/send flows, Leo billing/channel/session/message-log domain, landing page route split, Leo dashboard UI.

### Current Architecture
- No Léo-related backend or frontend files exist yet.
- The authenticated SPA root is still `/`; Phase 5 requires public `/` plus authenticated `/dashboard`.
- `Business` does not yet store any Léo add-on state.
- Dashboard API does not expose any Leo activity payload.

### Key Constraints
- Keep one channel per business, with delete-and-recreate for channel changes.
- Reuse existing auth/subscription conventions and keep all user-facing copy in French.
- Preserve modular monolith patterns: provider abstraction, API resources, form requests, small controllers.
- Deliver test-first where feasible, but phase scope requires batching by subsystem rather than single tiny commits.

### Open Questions
- None blocking enough to justify stopping. The phase spec is treated as the approved design baseline in autonomous mode.

### Implementation Order
1. Add plan doc and create feature branch.
2. Add business schema fields plus Leo tables/models/factories.
3. Add Leo contracts/services/tools and Telegram/Gemini infrastructure.
4. Add Leo API/webhook/controllers/middleware/resources/commands.
5. Add observer/job/throttle/message logging/dashboard activity backend wiring.
6. Add backend Pest tests and iterate until targeted suites pass.
7. Refactor router and auth redirections for public landing vs dashboard.
8. Implement Leo frontend types/api/composable/components/pages and dashboard widget.
9. Add Vitest coverage for landing/router/Leo flows.
10. Run full validation, perform adversarial review, resolve findings, update tracking docs, commit, push, create PR, watch CI.

### Acceptance Criteria
- [ ] Leo backend domain exists with one channel per business and message/session logging.
- [ ] Telegram webhook receives inbound messages, resolves business context, calls Gemini tools, and sends responses.
- [ ] Cancellation and no-show events dispatch Leo notifications with anti-loop throttling.
- [ ] Leo add-on billing activates/deactivates through Stripe subscription items and webhook failsafes.
- [ ] Dashboard exposes latest Leo activity and authenticated UI manages a single Leo channel.
- [ ] Public landing page is available at `/`; dashboard is moved to `/dashboard`; Leo management is on `/leo`.
- [ ] Backend and frontend tests cover the new flows well enough to support the phase exit criteria.

### Risks
- HIGH: Gemini and Telegram specs are only described in docs, so implementation details must be inferred conservatively.
- HIGH: Full 80% coverage gate for an entire new phase may require more tests than the minimum feature flow.
- MEDIUM: Stripe item sync logic can regress base subscription behavior if webhook branching is too broad.
- MEDIUM: Router refactor can break existing login/register/subscription navigation if redirects are not updated consistently.
- LOW: Landing page fidelity could drift from the template if not mirrored carefully.
