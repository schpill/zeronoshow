# Phase 3 APEX — Dashboard & Billing

## Analysis

### Project Structure

- `backend/` contient l'API Laravel 12, les jobs SMS, le middleware d'abonnement, les migrations Postgres et les tests PHPUnit/Pest.
- `frontend/` contient l'application Vue 3 + Pinia + Vitest, avec un dashboard minimal déjà connecté à `/api/v1/reservations`.
- `docs/phases/phase3.md` et `docs/dev/phase3.md` définissent précisément le périmètre Phase 3 et servent de design approuvé pour ce sprint.

### Existing Patterns

- Les endpoints API suivent le préfixe `/api/v1` et retournent des payloads JSON via resources Laravel.
- Le frontend s'appuie sur `apiClient` avec redirection automatique sur `401` et `402`.
- Les composants UI existants utilisent Tailwind, la typographie documentée dans `docs/graphics/polices.md`, et des variantes `dark:` déjà présentes sur une partie de l'interface.
- Le dashboard actuel réutilise `ReservationForm`, `ReservationList` et `ReservationRow`.

### Related Files

- Backend: `backend/routes/api.php`, `backend/routes/console.php`, `backend/app/Http/Controllers/Api/ReservationController.php`, `backend/app/Http/Middleware/RequireActiveSubscription.php`, `backend/app/Models/Business.php`
- Frontend: `frontend/src/pages/Dashboard.vue`, `frontend/src/pages/SubscriptionPage.vue`, `frontend/src/components/NavBar.vue`, `frontend/src/layouts/AppLayout.vue`, `frontend/src/composables/useReservations.ts`
- Specs: `docs/phases/phase3.md`, `docs/dev/phase3.md`, `docs/graphics/colors.md`, `docs/graphics/polices.md`

### Dependencies

- Déjà présentes: Laravel Sanctum, Redis-ready config, Vue 3, Pinia, Vitest.
- À ajouter ou câbler pour la phase: Stripe PHP SDK, Horizon, configuration mail exploitable pour les mails d'essai.

### Current Architecture

- Le middleware `RequireActiveSubscription` bloque actuellement tous les endpoints protégés si l'essai est expiré, ce qui ne respecte pas encore l'exigence "GET autorisé, POST bloqué".
- Le `ReservationController::index` joue déjà un rôle proche du dashboard, mais il manque les métriques de coût SMS mensuel, le taux no-show hebdo, et une API dédiée `/dashboard`.
- Le frontend a une page abonnement statique et aucun détail de réservation dédié, aucune logique de souscription, aucun toggle dark mode persistant.

### Key Constraints

- Respect strict du design system documenté en `docs/graphics/*`.
- TDD obligatoire: écrire les tests d'abord, vérifier l'échec, puis implémenter le minimum.
- Ne pas casser la phase 2: index/show/store/updateStatus, score de fiabilité, polling, cache dashboard existant.
- Le sprint couvre backend, frontend, scheduling, email, intégration Stripe et dark mode.

### Open Questions

- Les clés Stripe/SMTP locales pourront manquer, donc les tests devront utiliser des fakes/mocks robustes.
- L'installation Horizon/Stripe peut nécessiter un ajustement du lockfile et de la CI selon les dépendances réellement disponibles.

### Implementation Order

1. Écrire et faire échouer les tests backend pour dashboard, abonnement, webhooks, commandes.
2. Implémenter les services, contrôleurs, middleware et scheduling backend jusqu'au vert.
3. Écrire et faire échouer les tests frontend pour stats, navigation, bannière, détail, dark mode et dashboard.
4. Implémenter les composants/pages/composables frontend jusqu'au vert.
5. Exécuter lint, analyse statique, typecheck, tests backend/frontend, puis corriger.
6. Commit, push, PR et boucle CI.

### Acceptance Criteria

- [ ] `/api/v1/dashboard` retourne réservations, stats, coût SMS mensuel et taux no-show hebdo.
- [ ] `/api/v1/subscription` et `/api/v1/subscription/checkout` fonctionnent avec Stripe abstrait et testable.
- [ ] `/api/v1/webhooks/stripe` met à jour `subscription_status` de façon sûre.
- [ ] Les commandes d'email d'essai et de sync SMS mensuelle sont planifiées et testées.
- [ ] Le middleware n'autorise plus les mutations après expiration d'essai mais laisse les GET accessibles.
- [ ] Le dashboard frontend supporte vue jour/semaine, stats, bannière d'essai et insertion immédiate après création.
- [ ] La page détail réservation affiche les logs SMS et les informations client.
- [ ] Le tunnel d'abonnement frontend déclenche le checkout Stripe et reflète l'état courant.
- [ ] Le dark mode persiste dans `localStorage`, applique `dark` sur `<html>` et évite le flash initial.
- [ ] Les suites backend/frontend, lint et typecheck passent localement.

### Risks

- HIGH: installation ou API Stripe/Horizon divergente avec l'environnement local ou la CI.
- HIGH: volume du sprint important, nécessitant un découpage strict et des validations fréquentes.
- MEDIUM: dérive entre le design system doc et les composants si le refactor front est trop large.
- MEDIUM: cache dashboard et règles d'abonnement peuvent créer des régressions de phase 2.
- LOW: calculs date/semaine fuseau horaire entre frontend, backend et tests.
