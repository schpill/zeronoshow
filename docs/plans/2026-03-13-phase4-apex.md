# Phase 4 Hardening And Launch Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Deliver the full Phase 4 scope from `docs/phases/phase4.md` and `docs/dev/phase4.md`, implementing every repo-local hardening, UX, and deployment artifact while documenting and automating the externally dependent launch steps.

**Architecture:** Phase 4 extends the existing Laravel + Vue monorepo in four slices. Backend hardening adds named rate limiters, robust webhook verification, reservation edge-case handling, idempotent confirmation, health/smoke commands, purge automation, and production-safe JSON exception responses. Frontend hardening introduces a global toast/error/loading system, transport-level resilience for 429/503/network failures, Sentry bootstrap, and build optimizations. DevOps work is expressed as versioned deployment assets, workflows, and runbooks because actual provisioning, DNS, Sentry project creation, TLS issuance, and pilot rollout require external credentials and infrastructure.

**Tech Stack:** Laravel 12, PHP 8.3, Sanctum, Horizon, Redis, Twilio SDK, Sentry Laravel, Vue 3, TypeScript, Pinia, Vitest, Vite, Tailwind, GitHub Actions, Docker Compose, Nginx.

## Analysis

### Project Structure
- `backend/` contains the Laravel API, web confirmation flow, queue jobs, console routes, tests, Docker PHP/Nginx assets, and Horizon configuration.
- `frontend/` contains the Vue SPA, router, layout, composables, components, pages, and Vitest suite.
- `.github/workflows/` does not yet contain a production deploy workflow.
- Repo root currently has `docker-compose.yml` for local development but no production compose file.

### Existing Patterns
- Backend feature tests already cover auth, reservations, confirmation, webhooks, commands, and middleware with `RefreshDatabase`.
- Frontend tests use `@vue/test-utils` + `vitest`, with page/component tests colocated under `src/**/__tests__`.
- Controllers currently return mixed JSON and Blade responses without a shared API error envelope.
- Commands are defined in `backend/routes/console.php` instead of dedicated command classes, except planned stubs.

### Related Files
- Backend core: `backend/bootstrap/app.php`, `backend/routes/api.php`, `backend/routes/console.php`, `backend/app/Http/Controllers/Api/ReservationController.php`, `backend/app/Http/Controllers/ConfirmationController.php`, `backend/app/Http/Controllers/Webhook/TwilioWebhookController.php`, `backend/app/Services/TwilioSmsService.php`.
- Frontend core: `frontend/src/api/axios.ts`, `frontend/src/main.ts`, `frontend/src/App.vue`, `frontend/src/layouts/AppLayout.vue`, `frontend/src/pages/*.vue`, `frontend/vite.config.ts`.
- Specs: `docs/phases/phase4.md`, `docs/dev/phase4.md`, design references in `docs/graphics/*`.

### Dependencies
- `sentry/sentry-laravel` is already installed in Composer, but not wired in app bootstrap/config.
- Frontend does not yet depend on `@sentry/vue` or a compression plugin.
- Twilio SDK is present and currently validates against `Request::fullUrl()`, which is not sufficient for proxy/production reconstruction needs.

### Current Architecture
- Reservations are created in `ReservationController::store`, which currently only distinguishes verified phone numbers and "more than 30 minutes" scheduling.
- Confirmation links are single-use today because tokens are nulled after confirm/cancel.
- Twilio webhook handling delegates signature validation to `SmsServiceInterface`, then updates opt-out and `sms_logs`.
- App-wide SPA error handling is limited to 401/402 redirects in the Axios interceptor; no global user feedback or retry behavior exists.

### Key Constraints
- Manual actions requiring real external infrastructure cannot be completed from code alone: DNS cutover, DigitalOcean provisioning, Let's Encrypt issuance, UptimeRobot configuration, pilot onboarding, and real Sentry project validation.
- The user requested full autonomy with `-a -pr -ci`, so the implementation should proceed without further confirmation and create the PR/CI loop once code and repo-owned automation are ready.
- TDD applies to all behavioral changes: tests must be written/red before production edits for each slice.

### Scope Boundary
- Implement directly in repo: all backend and frontend code changes, tests, production compose/nginx/workflow/deploy scripts, env/runbook docs, PR, and CI watch.
- Deliver as documented automation/runbooks instead of live execution: DO droplet creation, GitHub secret population, DNS, TLS issuance, UptimeRobot/Sentry account setup, pilot onboarding.

### Acceptance Criteria
- [ ] Named rate limiters enforce the Phase 4 quotas and return `429` with `Retry-After`.
- [ ] Tampered Twilio webhook requests fail with `403` and valid requests still update logs/opt-outs.
- [ ] Reservation creation handles `<2h` and `<30min` edge cases exactly as specified.
- [ ] Duplicate confirmation/cancel requests are idempotent and terminal states return `410`.
- [ ] `/api/v1/health` exists and reports component status with 200/503 behavior.
- [ ] `sms-logs:purge` supports dry-run and real deletion for records older than 90 days.
- [ ] `smoke:test` validates critical app flow and is scriptable from CI/deploy.
- [ ] API exception responses are consistent JSON for common failures.
- [ ] Frontend exposes toast, loading, and retryable error states across key pages.
- [ ] Axios handles 429 with one GET retry and 503/network failures with persistent user feedback.
- [ ] Frontend build uses code splitting and compression plugin configuration.
- [ ] Production deploy assets and runbooks cover compose, nginx, workflow, env, backups, monitoring, and launch operations.
- [ ] PR is created and CI watch started after push.

### Risks
- HIGH: Phase 4 mixes code and live operations; completion must distinguish implemented automation from manual external execution.
- MEDIUM: Existing confirmation-token invalidation conflicts with idempotent duplicate handling and may require lookup logic beyond token-only state.
- MEDIUM: Frontend interceptor retry/toast behavior can destabilize tests if global state is not isolated.
- LOW: Production Docker workflow may need adjustment once real secrets and infrastructure naming are known.
