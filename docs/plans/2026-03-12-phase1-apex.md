# APEX Phase 1

## Analysis

### Project Structure

- Repo initially contained product specs, design system docs, and HTML templates only.
- `backend/` is now scaffolded with Laravel 12.
- `frontend/` is now scaffolded with Vue 3 + Router + Pinia + Vitest + ESLint + Prettier.
- No production application code existed before this execution.

### Existing Patterns

- Product requirements are fully specified in `docs/phases/phase1.md`.
- Task tracking is mirrored in `docs/dev/phase1.md`.
- Frontend design must follow `docs/graphics/colors.md`, `docs/graphics/polices.md`, and the mandatory HTML templates.
- Backend stack is Laravel + Sanctum + queued SMS + PostgreSQL/Redis.

### Related Files

- `docs/phases/phase1.md`
- `docs/dev/phase1.md`
- `docs/graphics/colors.md`
- `docs/graphics/polices.md`
- `docs/graphics/templates/template_backoffice_client.html`
- `docs/graphics/templates/template_site_vitrine.html`

### Dependencies

- Backend: Laravel 12, Sanctum, Twilio SDK, Pest, PHPStan/Larastan.
- Frontend: Vue 3, Vue Router, Pinia, Vitest, Tailwind CSS 3, Axios.
- Infra: Docker Compose, PostgreSQL 16, Redis 7, Mailpit, nginx.

### Current Architecture

- Monorepo root with separate `backend/` and `frontend/`.
- SPA frontend authenticates against Laravel API using bearer tokens.
- Public confirmation flow is rendered by Laravel Blade.
- SMS delivery is abstracted behind an interface and executed asynchronously by queue job.

### Key Constraints

- Implement all of Phase 1 autonomously.
- Use the design templates as visual authority for frontend rendering.
- Keep dark mode support in Vue components from the first implementation.
- Queue-based SMS only, never synchronous from controller code.
- Avoid unrelated refactors.

### Open Questions

- Specs conflict on some frontend implementation paths (`resources/js` vs `frontend/src`); implementation will keep the actual Vue scaffold under `frontend/src`.
- `Customer::getScoreTier()` spec text conflicts on null handling; API/UI will treat null as "no history"/`null`, not `at_risk`.
- Reservation "too late to confirm" behavior is underspecified; implementation will keep Phase 1 status set within existing enums and omit token generation for near-term bookings.

### Implementation Order

1. Finalize scaffolding and dependencies.
2. Write backend tests for auth, reservations, confirmation, middleware, and jobs.
3. Implement backend domain models, migrations, services, controllers, resources, and routes.
4. Write frontend tests for auth store, composables, badge, forms, and pages.
5. Implement frontend design system and SPA flows.
6. Add Docker, nginx, worker/scheduler, env defaults, and CI.
7. Run validation suites, perform adversarial review, and resolve defects.

### Acceptance Criteria

- [ ] Business registration/login/logout API works with Sanctum token responses.
- [ ] Reservation creation supports unverified and verified flows exactly as spec’d.
- [ ] Verification SMS is dispatched only through queue job and logged.
- [ ] Public confirmation/cancel flow updates reservation status and invalidates token.
- [ ] Customer lookup returns reliability data for the SPA.
- [ ] Vue SPA provides login/register/dashboard/reservation UX aligned with templates.
- [ ] Tailwind/fonts/colors/dark mode follow design references.
- [ ] Docker Compose and CI match the specified Phase 1 stack.

### Risks

- HIGH: Large surface area spanning backend, frontend, and infra in one pass.
- HIGH: Spec inconsistencies between implementation details and generated scaffold defaults.
- MEDIUM: Twilio behavior cannot be fully end-to-end verified without live credentials.
- MEDIUM: PostgreSQL-specific defaults must remain compatible with local test execution.
- LOW: Template-perfect frontend parity may require small follow-up polish after functional integration.
